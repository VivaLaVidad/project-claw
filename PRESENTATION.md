# Project Claw - 技术演讲稿

## 开场（2分钟）

各位好，我是 VivaLaVidad。今天要分享的是 **Project Claw**——一个工业级 AI 驱动的智能砍价系统。

这个项目解决的核心问题很简单：**如何让消费者和商家在线上实时砍价，就像在菜市场一样自然？**

传统电商是固定价格，没有讨价还价的空间。但在中国市场，尤其是生鲜、日用品这类高频消费品，消费者期望能砍价，商家也愿意根据库存、时段灵活定价。

Project Claw 就是这样一个平台：**C端消费者发起砍价意图 → 云端实时广播给所有在线商家 → 商家 AI 自动报价 → 消费者选择最优报价 → 秒级成交**。

整个过程完全自动化，没有人工介入。

## 架构设计（3分钟）

我们采用了**三端分离 + 云端中枢**的架构：

### 第一端：C端消费者（微信小程序 + Web）

消费者打开小程序，输入「我要买龙虾，预算 25 块」。系统立即生成一个 Intent（意图），发送到云端。

这个 Intent 包含：
- 商品名称
- 预算上限
- 消费者画像（价格敏感度、时间紧急度、品质偏好）

### 第二端：B端商家（本地 EdgeBox Agent）

每个商家在本地运行一个 Python Agent，连接到我们的云端 Hub。

当商家收到消费者的砍价意图时，Agent 会：
1. 查询本地菜单数据库获取底价
2. 调用 DeepSeek LLM 生成报价理由
3. 综合考虑库存、时段、消费者画像，生成最优报价
4. 秒级返回报价

这里的关键是**无状态设计**——每个 Agent 完全独立，可以水平扩展到 1000+ 个商家。

### 第三端：云端 Hub（Railway）

这是整个系统的大脑。用 FastAPI 实现，核心职责：

1. **意图广播**：收到 C端 Intent → 并发发送给所有在线 B端
2. **报价汇总**：收集所有 B端的报价 → 排序 → 返回给 C端
3. **成交下单**：C端选择报价 → Hub 下发成交指令给 B端
4. **审计日志**：所有交易事件实时推送给监控大屏

整个流程的延迟控制在 **< 3 秒**。

## 技术亮点（4分钟）

### 1. 异步并发架构

我们用 Python asyncio 实现了完全异步的 WebSocket 服务器。

为什么这很重要？因为传统的同步 I/O 模型，每个连接需要一个线程，1000 个商家就需要 1000 个线程，内存爆炸。

而 asyncio 可以用单线程处理 10000+ 个并发连接。我们的 Hub 在 Railway 免费套餐（512MB 内存）上稳定运行。

```python
# 核心代码示例
async def broadcast_intent(intent):
    merchants = list(self._merchants.values())  # 所有在线商家
    await asyncio.gather(
        *[m.send_json(intent) for m in merchants],
        return_exceptions=True
    )
```

### 2. 端到端加密通信

所有 B/C ↔ Hub 的消息都经过：
- **AES-GCM 加密**：防止中间人窃听
- **HMAC-SHA256 签名**：防止消息篡改
- **Nonce 重放保护**：防止重放攻击

这确保了交易的安全性。

### 3. 幂等性设计

每个 `/execute_trade` 请求都有唯一的 idempotency_key。

如果网络抖动导致请求重复，系统会自动去重，不会重复扣款。这是金融级别的可靠性要求。

### 4. AI 驱动的自动报价

这是最核心的创新。B端 Agent 不是简单的规则引擎，而是真正的 AI：

```python
# B端 Agent 的谈判逻辑
async def negotiate_intent(intent):
    # 1. 查询本地菜单获取底价
    bottom_price = self.store.query(intent.item_name)
    
    # 2. 调用 DeepSeek LLM 生成报价
    prompt = f"""
    消费者要买{intent.item_name}，预算{intent.expected_price}元。
    我们的底价是{bottom_price}元。
    请生成一个有竞争力的报价和销售话术。
    """
    response = await llm_client.call(prompt)
    
    # 3. 返回报价
    return MerchantOffer(
        offered_price=response.price,
        reply_text=response.pitch
    )
```

每个报价都是 AI 生成的，考虑了消费者画像、库存状态、时段因素。

### 5. 实时监控大屏

我们用 Streamlit 构建了一个「上帝视角」大屏，实时显示：

这个大屏通过 WebSocket 监听云端的审计事件流，延迟 < 100ms。

## 技术挑战与解决方案（2分钟）

### 挑战 1：如何在 3 秒内汇总 100+ 个商家的报价？

**解决方案**：
- 异步并发发送意图（不等待）
- 设置 3 秒超时，部分报价也返回
- 优先返回高分报价（不需要等所有商家）

### 挑战 2：如何保证交易的一致性？

**解决方案**：
- 幂等性设计（idempotency_key）
- Redis 分布式锁
- 订单状态机（created → broadcasted → executing → executed）

### 挑战 3：如何处理网络抖动和 Agent 离线？

**解决方案**：
- 心跳检测（20 秒 ping 一次）
- 自动重连（指数退避）
- 降级方案（后端离线时用本地 mock 数据）

### 挑战 4：如何在 Railway 免费套餐上运行？

**解决方案**：
- 精简依赖（去掉 torch、easyocr 等重包）
- 无状态设计（可随时重启）
- 流式处理（不缓存大数据）

## Q&A 预案

**Q: 为什么选择 Python 而不是 Go/Rust？**

A: Python 的生态最成熟（FastAPI、asyncio、Pydantic）。对于 I/O 密集型应用，Python asyncio 的性能完全够用。而且团队熟悉 Python，开发效率更高。

**Q: 如何处理商家恶意报价（比如报价 1 块钱）？**

A: 有两层防护：
1. 商家画像中有「底价」限制，Agent 不会低于底价报价
2. Hub 端有价格合理性检查，异常报价会被过滤

**Q: 如何扩展到 10000+ 商家？**

A: 
1. 无状态设计，可以水平扩展 Hub（多个 uvicorn workers）
2. Redis 作为分布式状态存储
3. 数据库分片（按 merchant_id 分片）

**Q: 成本是多少？**

A: 
- Railway 云端：$5/月（免费套餐）
- Streamlit Cloud：免费
- 本地 B端：只需要一台电脑（可以是树莓派）
- 总成本：< $10/月

**Q: 如何保证数据隐私？**

A: 
- 所有通信都是 AES-GCM 加密
- 消费者数据不存储在云端（只在本地小程序）
- 商家数据只有商家自己能看到
